# Выводы по генерации 50 заявок на ДПО

## Борьба с mode collapse

Генерация выполнена через DeepSeek V4 (`deepseek-chat`, `https://api.deepseek.com`) с использованием кастомного `llm_client.py` (JSON-mode + Pydantic-валидация).

**Города** (12 вариантов) — циклический `seed_city` в промпте: `CITIES[i % 12]` для каждого i-го вызова. Итог: каждый город получил 4-5 заявок (по 8-10%), ни один не превысил 40%.

**Специальности** (12 вариантов) — на первом прогоне DeepSeek выдал 44% «Инженер-строитель». Решение: добавили циклический `seed_speciality = SEED_SPECIALITIES[i % len(SEED_SPECIALITIES)]` с явной инструкцией в промпте. Второй прогон: каждая специальность ровно по 5 заявок (по 10%), ни одна не превысила 35%.

**Курсы** — без принудительного seed, распределение 4-8 заявок на курс, приемлемо.

**Оставшаяся проблема: коллапс имён.** DeepSeek упорно генерирует одни и те же ФИО: «Анна Сергеевна Кузнецова» (5 раз), «Иванова Анна Сергеевна» (5 раз), «Сергей Владимирович Кузнецов» (4 раза). Промпт содержал инструкцию «Разнообразь ФИО — не повторяй одни и те же имена», но модель её игнорирует. Это системное ограничение конкретной модели — DeepSeek V4 Flash (используемый OpenClaw) в JSON-mode склонен к повторению паттернов из обучающих данных, где фамилии Кузнецов, Иванов, Смирнов, Петров — самые частотные. Решение: потребовалась бы или `frequency_penalty`, или постинспекция с регенерацией дубликатов, или передача уже использованных имён в промпт (context window позволяет).

## Работа @field_validator

Валидатор `check_graduation_age_consistency` проверяет, что год окончания вуза (`graduation_year`) соответствует возрасту (`age`). Логика: возраст при выпуске = `graduation_year - (2025 - age)`. Допустимый диапазон 17–40 лет.

**Тест с заведомо некорректными данными** (`age=30, graduation_year=1985` → возраст при выпуске -10 лет) — валидатор успешно отловил ошибку с сообщением «слишком ранний год окончания».

Однако в реальной генерации через DeepSeek с `response_format={"type": "json_object"}` и Pydantic-валидацией **ни одного конфликта не возникло** — модель ни разу не сгенерировала противоречивую пару (age, graduation_year). Возможно, это связано с JSON-mode: модель вынуждена строго следовать схеме, и числовые поля заполняются консистентно. При работе через свободный текстовый режим (без схемы) вероятность ошибки выше, и валидатор стал бы критичен.
