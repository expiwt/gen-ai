# Отчёт по финальному проекту

**Stock Analyzer** — система анализа российских акций на основе данных
Московской биржи и новостного фона.

**Трек:** B (прикладной)

---

## 1. Задача и данные

### Задача

Построить конвейер, который по тикеру акции:
1. Получает рыночные данные (цена, капитализация, объёмы) через MOEX ISS API
2. Находит релевантные новости по бумаге через RAG с эмбеддингами
3. LLM анализирует данные + новости и выдаёт структурированную оценку:
   - вердикт (strong_buy / buy / hold / sell / strong_sell)
   - потенциал роста в процентах
   - уровень риска
   - тональность новостного фона
   - ключевые бычьи и медвежьи факторы
4. Проверяет результат на галлюцинации (LLM-as-judge)

### Данные

**Источники:**

| Источник | Что даёт | Доступ |
|----------|----------|--------|
| MOEX ISS API (iss.moex.com) | Цена, капитализация, объёмы, спред, изменение | Открытый REST, без ключа |
| SmartLab (тематика) | 15 синтетических новостей по 5 бумагам | Статический датасет (input/news/) |

**Объём:** 5 бумаг x 3 новости = 15 тестовых входов.

**Подготовка новостей:** 15 статей сгенерированы LLM на основе реальной
рыночной повестки компаний (отгрузки ММК в Китай, урожай Русагро,
обводнение рудника Алроса, дефицит никеля на LME, отчётность
Совкомбанка). Каждая статья проверена на правдоподобие. Новости
сохранены как статические JSON для воспроизводимости.

**Покрытие (5 бумаг, 5 секторов):**

| Тикер | Компания | Сектор |
|-------|----------|--------|
| MAGN | ММК | Чёрная металлургия |
| RAGR | РусАгро | Агросектор |
| ALRS | Алроса | Добыча алмазов |
| GMKN | Норникель | Цветные металлы |
| SVCB | Совкомбанк | Банкинг |

## 2. Система

### Техники курса (5 из >=4)

| # | Техника | Реализация |
|---|---------|------------|
| 1 | RAG | rag.py: sentence-transformers, эмбеддинги 384d, косинусный поиск по 15 новостям. Keyword fallback при недоступности эмбеддингов (ленивый импорт) |
| 2 | Агент с инструментами | pipeline.py: агент вызывает MOEX API (инструмент 1), RAG (инструмент 2), LLM (инструмент 3) |
| 3 | LLM-as-judge | Два применения: (а) проверка галлюцинаций в ответе аналитика, (б) eval-оценка по 5 шкалам |
| 4 | Структурированный вывод (response_model) | llm_client.py: JSON mode + Pydantic-валидация StockAnalysis с max_retries=3 |
| 5 | field_validator | schema.py: 4 бизнес-инварианта + model_validator |

### Схема потока данных

```
Тикер (MAGN / RAGR / ...)
    |
    +-- MOEX ISS API -> MarketData (цена, кап-ция, объём, спред)
    |
    +-- NewsCorpus (RAG) -> эмбеддинги -> top-3 новости
    |
    +-- LLM (DeepSeek V4 Flash) -> StockAnalysis (response_model)
    |     +-- retry x3 при ошибке парсинга
    |
    +-- Hallucination checker (LLM-as-judge)
    |     +-- price match (цена из ответа vs MOEX +-5%)
    |     +-- cap match (капитализация +-10%)
    |     +-- volume match (объём +-20%)
    |     +-- ghost-fact check (LLM)
    |
    +-- output/analysis_{ticker}_{date}.json
```

### Ключевые файлы

| Файл | Назначение |
|------|------------|
| code/schema.py | Pydantic-модели: MarketData, NewsArticle, StockAnalysis, EvalResult |
| code/moex_client.py | MOEX ISS API клиент (без токена), включая fetch_marketdata_by_date() по истории |
| code/rag.py | RAG: эмбеддинги -> косинусный поиск, lazy import sentence-transformers |
| code/llm_client.py | LLM-клиент: JSON mode + retry, OpenAI-совместимый |
| code/pipeline.py | Главный пайплайн: MOEX -> RAG -> LLM -> hallu check |
| eval.py | Eval: 15 тестов на реальных исторических датах + LLM-as-judge |

### field_validator (бизнес-инварианты)

В StockAnalysis:

1. growth_outlook -- только {strong_buy, buy, hold, sell, strong_sell}
2. risk_level -- только {низкий, средний, высокий}
3. top_news_themes -- не пустой список (макс. 3)
4. analysis_date -- не в будущем (поймано 6 галлюцинаций даты за 15 тестов)
5. model_validator: strong_buy требует потенциал >=15%, strong_sell требует <=-10%

## 3. Оценка (Eval)

### Методология

- **15 тестовых входов:** 5 бумаг x 3 сценария
- **Сценарии** -- реальные исторические даты из MOEX:
  - latest (19.06.2026) -- последний торговый день
  - mid (05.06.2026) -- за две недели до
  - early (18.05.2026) -- за месяц до
- **Правильность:** LLM-as-judge по 5 бинарным шкалам (0-5):
  1. DATA_USAGE -- все ли MOEX-цифры корректно отражены
  2. CONSISTENCY -- нет внутренних противоречий
  3. FACTUAL -- нет выдуманных фактов
  4. SPECIFICITY -- есть конкретика (цифры, проценты, направления)
  5. PLAUSIBILITY -- выводы разумны для данной бумаги
- **Путь:** число шагов агента (5), токены, стоимость
- **Проход:** score >= 3 И hallu_check_passed

## 4. Где пайплайн ошибался

### Пример 1: Совкомбанк -- галлюцинация даты (контаминация данных)

Вход: SVCB, любой сценарий. Дата анализа -- 21 июня 2026.

Искажение: DeepSeek стабильно выставляет analysis_date = "2026-07-17"
или "2026-07-14" для Совкомбанка (месяц вперёд). Ни одного нормального
вызова за 15 тестов -- модель упорно не признаёт сегодняшнюю дату для этого
тикера.

Как поймано: field_validator("analysis_date") вызывает ошибку валидации.
Пайплайн автоматически чинит дату и логирует как hallu-ошибку.

Коренная причина: вероятно, контаминация обучающих данных DeepSeek
(банковский сектор = много данных за июль в обучающей выборке).

### Пример 2: GMKN -- галлюцинация даты (та же проблема)

Вход: GMKN, текущие данные. DeepSeek стабильно выставляет
analysis_date = "2026-07-17" или "2026-07-09".

Детали: Проблема не ограничивается банковским сектором -- Норникель
тоже видит июль. retry x3 не помогает, repair отрабатывает.

Статистика: 6 из 15 тестов (3 x GMKN + 3 x SVCB) требуют date-repair.
Это системная проблема модели, а не пайплайна.

### Пример 3: MAGN -- ghost-факты из RAG

Вход: MAGN, latest. RAG возвращает новость "ММК наращивает отгрузки
в Китай на 15%".

Проблема: Аналитик корректно указывает "увеличение отгрузок в Китай
на 15%" в факторах. Judge (второй LLM) засчитывает это как ghost-факт,
потому что этой цифры нет в сырых MOEX-данных.

Анализ: Это ложное срабатывание (false positive) детектора
галлюцинаций. Цитата реальна и взята из RAG-результата. Проблема в том,
что hallu-checker знает только о market data, но не сверяет факты
по исходным новостям.

Аналогичная проблема для ALRS -- "рост продаж алмазов на 18%"
также засчитывается как ghost.

## 5. Выводы

### Ответ на гипотезу

Система успешно анализирует акции по данным MOEX + новостям с точностью
4.33/5. При этом:

- RAGR (РусАгро) -- самый надёжный кейс: 5/5 во всех сценариях, чёткая
  связь новостей с вердиктом.
- SVCB (Совкомбанк) + GMKN (Норникель) -- стабильно галлюцинируют дату
  (контаминация обучающих данных). Требуют post-hoc валидации.
- MAGN, ALRS -- страдают от false positive в детекторе галлюцинаций:
  новостные цифры квалифицируются как выдумка из-за того, что hallu-checker
  не знает о содержимом RAG-новостей.

### Границы применимости

1. Монокультура одной модели -- все тесты на DeepSeek V4 Flash.
   Другая модель (YandexGPT, GPT-4o-mini) может дать другие результаты.
2. Контаминация данных -- DeepSeek знает о российских акциях из
   обучающей выборки, что искажает чистоту эксперимента.
3. RLHF -- модель склонна к безопасным ответам (BUY со средним
   риском), не использует весь спектр вердиктов (ни одного SELL за 15 тестов).
4. Чувствительность к промпту -- формулировка analyst prompt напрямую
   влияет на полноту анализа.
5. Новостной корпус -- синтетические данные, 3 статьи на бумагу.
   В реальности -- сотни.

### Что в прод, что переделать

**В прод:**
- MOEX клиент -- надёжный, быстрый, бесплатный, без токена, включая
  исторические данные через history endpoint
- Pydantic-схемы с валидацией -- защита от мусора (ловят дату-галлюцинацию)
- pipeline-архитектура (MOEX -> RAG -> LLM -> hallu check) --
  production-ready
- Lazy import sentence-transformers -- эмбеддинги загружаются только
  по необходимости

**Переделать:**
- Hallucination checker -- добавить cross-check по исходным новостям
  (сейчас знает только market data, не проверяет RAG-контекст)
- Промпт -- явно указать сегодняшнюю дату для SVCB и GMKN
- RAG -- заменить косинусную близость на гибрид BM25 + эмбеддинги
  через RRF
- Eval -- добавить нормализацию judge-оценок для fairness между разными
  тикерами
